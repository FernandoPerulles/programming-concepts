Master Reference

Data Structures

A data structure is a way of organizing data in memory so that it can be used efficiently. The word "efficiently" is the key — it means performing operations (access, search, insert, delete) with the least amount of time and memory. Every data structure is a trade-off: fast at some things, slow at others. Understanding those trade-offs is the entire game.

This reference covers the seven most fundamental data structures, with crystal-clear conceptual explanations, visual models, step-by-step operation traces, and the Big O complexities you need to make professional decisions.

Structures covered: 7
Big O analysis: included
Level: beginner → intermediate
Mental Model
Think of data structures like physical containers in a warehouse. An array is a numbered shelf — you can walk straight to position 7. A linked list is a scavenger hunt — each clue tells you where to find the next one. A hash table is a magic filing cabinet — you tell it the name and it instantly opens the right drawer. Each is useful, each has limits.
Foundation

Big O Notation

Before studying any data structure, you must understand Big O. It is the universal language programmers use to describe how an algorithm's performance scales as the size of the input grows. We denote input size as n.

Big O describes the worst-case scenario and ignores constants — we don't care if something takes 2n or 5n steps; both are O(n). We care about the shape of the curve, not the exact number.

Important Nuance
Big O measures time complexity (how long) and space complexity (how much memory) separately. A data structure might be O(1) to insert but O(n) in space. Always consider both dimensions.

How to Read It

When you see O(log n) for a binary search tree lookup, it means: if your tree has 1,000 nodes, you need at most ~10 comparisons to find any element. If you double the tree to 2,000 nodes, you only need ~11 comparisons. The performance degrades incredibly slowly — this is the power of logarithmic algorithms.

When you see O(n) for a linked list search, it means: with 1,000 elements, you may need to check all 1,000. With 2,000, you may check all 2,000. It scales directly with the data size.

Structure 01

Arrays

An array is a collection of elements stored in contiguous (adjacent) memory locations. This is the single most important fact about arrays. Because every element is the same size and they sit next to each other in memory, the CPU can jump directly to any element using simple arithmetic:

// address of element at index i:
address = base_address + (i × element_size)

// Example: int array starts at address 1000, each int = 4 bytes
arr[0] → 1000  // base
arr[1] → 1004  // +4
arr[4] → 1016  // +16

This is why array index access is O(1) — constant time, regardless of array size. The CPU doesn't scan; it computes.

Static vs. Dynamic Arrays

Static arrays have a fixed size declared at creation time (C, C++). If you need more space, you manually allocate a new array and copy everything. Dynamic arrays handle this automatically (Python lists, Java ArrayList, JavaScript arrays). Under the hood, they still use a contiguous block but over-allocate by roughly 2× when they run out of room, making amortized insertion O(1).

Why Insertion is O(n)
To insert an element at index 2 in an array of 1,000 elements, you must shift every element from index 2 to 999 one position to the right to make room. That's up to n operations — hence O(n). Appending to the end is O(1) (amortized) because no shifting is needed.

You need fast random access by index. You're storing a fixed or slowly-changing collection. Your primary operations are reads, not inserts/deletes in the middle. Examples: pixel buffers, lookup tables, storing a fixed-size dataset, implementing other data structures internally.

Linked Lists

A linked list stores elements in nodes scattered anywhere in memory. Each node holds two things: its data, and a pointer (reference) to the next node. Unlike arrays, there is no contiguous block — nodes are connected by pointers forming a chain.

The Critical Trade-off
When people say "insertion in a linked list is O(1)", they mean the actual pointer-swapping operation is O(1) — once you have the node you want to insert after. But finding that position requires O(n) traversal from the head. The full story is: O(n) to find + O(1) to insert. Always specify which part you mean.

Why No Random Access?

Nodes are scattered across memory with no mathematical relationship between addresses. To get element at index 4, you must follow the chain: head → node1 → node2 → node3 → node4. That's O(n) — you can't compute the address directly like arrays can.

Structure 03

Stacks

A stack is an abstract data type (ADT) that enforces a LIFO (Last In, First Out) ordering. Think of a stack of plates — you can only put a plate on top or take one off the top. You have no access to plates in the middle.

A stack has exactly three fundamental operations:

  • push(x) — add element x to the top
  • pop() — remove and return the top element
  • peek() — return the top element without removing it

Implementation
A stack is an ADT — it describes behavior, not implementation. You can build one with an array (track a "top" index) or a linked list (always insert/delete from head). The choice affects memory usage but not the O(1) complexity of push/pop.

Real-World Uses of Stacks
Function call stack (every function call pushes a frame; return pops it). Undo/redo in text editors. Browser back button. Expression parsing and evaluation (e.g., converting infix to postfix notation). Depth-first search (DFS) in graphs. Balanced parentheses checking.
Structure 04

Queues

A queue is an ADT that enforces FIFO (First In, First Out) ordering. Think of a checkout line — the first person in line is the first to be served. New people join at the back; service happens at the front.

The fundamental operations:

  • enqueue(x) — add element x to the back
  • dequeue() — remove and return the front element
  • peek() — return the front element without removing it

Variants

Priority Queue — elements are dequeued by priority, not arrival order. Usually implemented with a heap. Used in Dijkstra's shortest path, job schedulers, and A* pathfinding.

Deque (double-ended queue) — supports enqueue and dequeue at both ends. Useful for sliding window problems and implementing both stacks and queues.

Circular Queue — a fixed-size queue implemented with an array where the back wraps around to the front. Avoids wasted space after dequeuing.

Real-World Uses of Queues
CPU task scheduling. Print spooler. BFS (breadth-first search) in graphs. Event loops (JavaScript's event queue). Network packet buffering. Rate limiting and request queues in web servers.
Structure 05

Trees

A tree is a hierarchical, non-linear data structure of nodes connected by edges. Unlike arrays or linked lists which are linear (one element after another), a tree branches. Key vocabulary you must know:

  • Root — the top-most node with no parent
  • Node — each element in the tree
  • Edge — the connection between parent and child
  • Leaf — a node with no children
  • Height — longest path from root to a leaf
  • Depth — distance of a node from the root
  • Subtree — a node and all its descendants

Binary Search Tree (BST)

The most fundamental tree type. Every node has at most two children. The critical BST invariant (rule) is:

The BST Rule — memorize this
For every node N: all values in the left subtree are less than N, and all values in the right subtree are greater than N. This rule must hold for every node, not just the root.

BST Visual — nodes [4, 2, 6, 1, 3, 5, 7]

How BST Search Works — why O(log n)?

Searching for 5 in the tree above: start at root (4). Is 5 > 4? Yes → go right to 6. Is 5 < 6? Yes → go left to 5. Found! We only checked 3 nodes out of 7 total. Each comparison eliminates half the tree, giving O(log n) — identical to binary search on a sorted array.

Balanced vs. Degenerate BST
The O(log n) guarantee only holds when the tree is balanced — roughly the same number of nodes on each side. If you insert sorted data [1, 2, 3, 4, 5] into a plain BST, you get a degenerate tree (essentially a linked list) with O(n) search. This is why self-balancing trees (AVL trees, Red-Black trees) exist — they automatically rebalance after every insert/delete to preserve O(log n).

Tree Traversals

There are three standard ways to visit every node in a BST:

// In-order (Left → Root → Right) → produces SORTED output: 1,2,3,4,5,6,7
// Pre-order  (Root → Left → Right) → useful for copying a tree: 4,2,1,3,6,5,7
// Post-order (Left → Right → Root) → useful for deletion: 1,3,2,5,7,6,4

Other Important Tree Types

Heap — a complete binary tree where every parent is greater (max-heap) or smaller (min-heap) than its children. Powers priority queues. O(log n) insert/delete.

AVL Tree / Red-Black Tree — self-balancing BSTs. Used in Java's TreeMap, Linux kernel scheduling, database indexes.

Trie (prefix tree) — each node represents a character. Used for autocomplete, spell-checkers, and IP routing. Lookups are O(m) where m is string length — independent of how many strings are stored.

B-Tree — generalizes BSTs to have more than 2 children. Optimized for disk access. Used in virtually every relational database index (MySQL, PostgreSQL).

Structure 06

Hash Tables

A hash table (also called a hash map or dictionary) maps keys to values with near-instant lookup. It achieves this using a hash function — a function that converts any key (string, number, object) into an integer index, which determines where in an array the value is stored.

// Conceptually:
index = hash_function(key) % array_size

// Example: array_size = 7
hash("name")  = 3  → store at index 3
hash("age")   = 1  → store at index 1
hash("city")  = 5  → store at index 5

// Lookup is then O(1): compute hash → jump to index → done

Collisions — the critical concept

A collision occurs when two different keys hash to the same index. This is unavoidable (pigeonhole principle: if you have more possible keys than buckets, collisions must occur). Managing collisions is the most important implementation detail of hash tables.

Never Skip This
Any textbook or tutorial that explains hash tables without explaining collision resolution is giving you an incomplete picture. Collisions directly affect the worst-case performance — a poorly implemented hash table can degrade from O(1) to O(n).

Strategy 1 — Chaining

Each bucket holds a linked list of all key-value pairs that hash to that index. On collision, simply append to the list.

0
— empty —
1
"age"→28 → "zip"→10001 → NULL
2
— empty —
3
"name"→"Alice" → "pet"→"cat" → NULL  (collision!)

Chaining is simple and handles high load well. Worst case: if every key hashes to the same bucket, lookup becomes O(n). In practice, with a good hash function and load factor below ~0.7, performance stays near O(1).

Strategy 2 — Open Addressing

All data stays in the main array — no linked lists. On collision, probe for the next available slot according to a rule:

  • Linear probing — check index+1, index+2, index+3, etc.
  • Quadratic probing — check index+1, index+4, index+9, etc.
  • Double hashing — use a second hash function to determine step size

Load Factor

The load factor (λ) = number of entries / number of buckets. As λ rises, collisions increase and performance degrades. Most hash table implementations automatically rehash (create a larger array and re-insert everything) when λ exceeds ~0.7 for open addressing or ~1.0 for chaining.

Use Hash Tables When
You need O(1) average-case lookups by a key. Counting frequencies (word counts, character counts). Caching (memoization). Implementing sets. Detecting duplicates. Hash tables power Python dicts, JavaScript objects, Java HashMaps — they're everywhere.
Structure 07

Graphs

A graph is a collection of vertices (nodes) connected by edges. Trees are a special case of graphs (a connected, acyclic graph). Graphs are arguably the most powerful and widely applicable data structure — social networks, GPS maps, the internet, and dependency resolvers are all graphs.

Key Terminology

  • Directed graph (digraph) — edges have direction (A→B doesn't mean B→A). Example: Twitter follows.
  • Undirected graph — edges are bidirectional. Example: Facebook friendships.
  • Weighted graph — edges have a numeric weight/cost. Example: road distances.
  • Degree — number of edges connected to a vertex.
  • Path — a sequence of vertices connected by edges.
  • Cycle — a path that starts and ends at the same vertex.
  • Connected graph — every vertex is reachable from every other vertex.

Visual — Weighted Undirected Graph

Graph Representations in Code

There are two main ways to store a graph in memory:

Adjacency List — each vertex stores a list of its neighbors. Space efficient for sparse graphs (few edges). O(V+E) space.

// Adjacency list for the graph above
graph = {
  "A": [("B", 4), ("D", 2), ("E", 7)],
  "B": [("A", 4), ("C", 3), ("E", 5)],
  "C": [("B", 3), ("F", 6)],
  "D": [("A", 2), ("E", 3)],
  "E": [("A", 7), ("B", 5), ("D", 3), ("F", 8)],
  "F": [("C", 6), ("E", 8)]
}

Adjacency Matrix — a V×V grid where matrix[i][j] = weight of edge from i to j (or 0/infinity if no edge). O(V²) space. Fast to check if two nodes are connected: O(1). Bad for sparse graphs.

Graph Traversal Algorithms

BFS (Breadth-First Search) — uses a queue. Explores all neighbors at the current depth before going deeper. Finds the shortest path in unweighted graphs. O(V+E).

DFS (Depth-First Search) — uses a stack (or recursion). Explores as far as possible down one path before backtracking. Used for cycle detection, topological sorting, maze solving. O(V+E).

Dijkstra's Algorithm — finds the shortest path in a weighted graph. Uses a priority queue. O((V+E) log V) with a binary heap.

Real-World Uses of Graphs
GPS navigation (shortest path). Social networks (connections, mutual friends). Web crawlers (the entire internet is a directed graph). Dependency resolution (npm, pip). Compilers (detecting circular imports). Recommendation engines. Flight route optimization.

Decision Guide

Choosing the right data structure is a practiced skill. Use these rules of thumb — then always verify with concrete complexity analysis for your specific access patterns.

→ Use an Array
Fast access by position
You need O(1) lookup by index. Data is fixed-size or mostly appended. You're iterating sequentially (great cache locality).
→ Use a Linked List
Frequent insert/delete at ends
You insert and delete constantly at head/tail. Size is highly dynamic. You don't need random access. Building a queue or deque.
→ Use a Stack
LIFO ordering needed
Function call management, undo operations, DFS traversal, balanced bracket checking, expression evaluation.
→ Use a Queue
FIFO ordering needed
Task scheduling, BFS traversal, message buffers, rate limiting, event processing in order of arrival.
→ Use a BST / Sorted Tree
Sorted data + fast search
You need data kept in sorted order. You need O(log n) insert, delete, and range queries. Database indexes. Use self-balancing variants (AVL, Red-Black) in production.
→ Use a Hash Table
O(1) key-based lookup
Lookup by a unique key (username, ID). Frequency counting. Caching results. Deduplication. When order doesn't matter and you need raw speed.
→ Use a Graph
Relationships between entities
Entities have complex many-to-many relationships. You need pathfinding, reachability, or cycle detection. Social networks, routing, dependency graphs.
→ Use a Heap
Repeated min/max access
You always need the smallest or largest element. Priority queues, scheduling, Dijkstra's algorithm, top-K problems, median maintenance.
The Final Word
No data structure is universally best. The art of software engineering is knowing their trade-offs so deeply that you can look at a problem and immediately see its shape — "this is a priority queue problem", "this screams hash table", "this graph is begging for Dijkstra". That instinct comes from studying and using each structure until the Big O numbers feel like muscle memory.
